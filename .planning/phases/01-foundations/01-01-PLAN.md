<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer machine → git repository | Pre-commit/pre-push hooks guard against unintended secrets or broken code reaching git |
| developer machine → CI runner | CI workflow validates code on PR before merge; no secrets exposed in logs |
| `.env.example` → real `.env` | `.env.example` documents shape only; `.gitignore` MUST prevent real `.env` from being committed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | `.env` file with real secrets accidentally committed | mitigate | `.gitignore` includes `.env`, `.env.local`, `.env.*.local`; only `.env.example` (placeholders) is tracked |
| T-01-02 | Tampering | npm/Go/external installs introduced via this plan | mitigate | All npm installs pull from the `[ASSUMED]` list in 01-RESEARCH.md §Package Legitimacy Audit; no new external dependencies introduced beyond that audited list |
| T-01-SC | Tampering | npm installs in this plan | mitigate | Per 01-RESEARCH.md §Package Legitimacy Audit, all packages in this plan are `[ASSUMED]` (slopcheck unavailable). The audit table in 01-RESEARCH.md lists every package by name + version + source repo; install must use those exact versions; if any new package is needed beyond the list, halt and add to audit table first |
| T-01-03 | Tampering | Pre-commit hooks bypassed via `--no-verify` | accept | Solo developer; documented in CLAUDE.md as a discipline rule, not a technical control. Pre-push hook remains second line of defense. |
| T-01-04 | Denial of Service | Husky / lint-staged misconfiguration blocks all commits | mitigate | Pre-commit hook fails fast with clear error message; documented bypass for emergencies (`git commit --no-verify`) referenced only in CLAUDE.md, not in default workflow |

</threat_model>
