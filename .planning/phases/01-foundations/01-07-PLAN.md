<objective>
Plug the Claude Code hook surface into the daemon pipeline built in plan 06. Three components: (1) a tiny Go shim that Claude Code invokes per hook fire (cold-start budget ≤15ms), (2) a daemon-side loopback HTTP bridge that authenticates the shim and hands payloads to the (3) Claude Code adapter, which normalises each of the 6 Claude Code hooks (D-22) into a CanonicalEvent. ## Shim contract (shim/main.go ~40-60 LOC)

  Inputs:
    - stdin: hook payload JSON (Claude Code emits this on every hook fire)
    - env: FENNEC_SHIM_SECRET, FENNEC_DAEMON_PORT (default "7821")
  Behavior:
    - Read all of stdin (single Read, no streaming).
    - HTTP POST to http://127.0.0.1:${FENNEC_DAEMON_PORT}/v1/hook with body=stdin payload AND header X-Fennec-Shim-Secret: ${FENNEC_SHIM_SECRET} AND Content-Type: application/json
    - Client timeout: 15ms total (deadline including connect + write + read)
    - On any error (timeout, daemon down, connection refused): exit 0 (FAIL-OPEN per D-23)
    - On success: exit 0
    - NEVER write to stderr (Claude Code might surface it to the user; fennec must be invisible)

  Cross-compile targets (Makefile):
    darwin/arm64 — Apple Silicon Macs
    darwin/amd64 — Intel Macs
    linux/amd64 — Phase 5 only, but built today for the Makefile completeness
    windows/amd64 — Phase 5 only

## Loopback bridge contract (daemon/src/adapters/loopback-bridge/server.ts)

  HTTP server on http.createServer(...).listen(PORT, "127.0.0.1") — NEVER 0.0.0.0
  PORT from env.FENNEC_DAEMON_PORT (default 7821)
  Routes:
    POST /v1/hook
      Validate X-Fennec-Shim-Secret header matches env.FENNEC_SHIM_SECRET. Mismatch → 401, log "rejected-loopback-attempt" + sourceAddr.
      Parse JSON body.
      Emit "hook" event to listeners (EventEmitter-style) with the parsed payload.
      Return 202 within 5ms (so shim's 15ms total budget holds).
    GET /v1/health
      Return 200 { status: "ok" }.

  Shim secret loaded from /etc/fennec/shim-secret (mode 0644 — world-readable so the user-context shim can read it; cf. Pattern 9 threat model — this is acceptable because loopback is not network-exposed and same-UID processes could already write directly to the queue).

## Claude Code adapter (daemon/src/adapters/claude-code/adapter.ts)

  Implements Adapter from daemon/src/adapters/adapter.ts:
    tool = "claude-code"
    version = "0.1.0"
    start(emit): subscribe to loopback bridge "hook" events; for each payload call normalizeHookPayload + emit
    stop(): unsubscribe

## Payload normaliser (daemon/src/adapters/claude-code/payload-normaliser.ts)

  Anthropic Claude Code hook payload shape (per https://code.claude.com/docs/en/hooks):
    {
      session_id: string,
      transcript_path: string,
      cwd: string,
      hook_event_name: "UserPromptSubmit" | "PostToolUse" | "SessionStart" | "SessionEnd" | "PreCompact" | "SubagentStop",
      // For UserPromptSubmit:
      prompt?: string,
      // For PostToolUse:
      tool?: { name: string, response?: { usage?: { input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens } } },
      // For SessionStart/SessionEnd/PreCompact/SubagentStop: minimal payload
    }

  normalizeHookPayload(raw) → buildCanonicalEvent input:
    {
      tool: "claude-code",
      adapter_version: "0.1.0",
      kind: map(hook_event_name) → EventKind ("UserPromptSubmit" → "prompt_submitted", "PostToolUse" → "tool_call", "SessionStart" → "session_start", etc.)
      payload: {
        prompt_text: raw.prompt ?? "",  // POST-redaction happens downstream
        session_id: raw.session_id,
        cwd: raw.cwd,
        hook_event: raw.hook_event_name,
        usage: raw.tool?.response?.usage ? {
          input_tokens: raw.tool.response.usage.input_tokens,
          output_tokens: raw.tool.response.usage.output_tokens,
          cache_creation_input_tokens: raw.tool.response.usage.cache_creation_input_tokens,
          cache_read_input_tokens: raw.tool.response.usage.cache_read_input_tokens,
        } : undefined,
      },
      occurred_at: undefined,  // buildCanonicalEvent stamps with now()
    }

## Managed-settings install (daemon/src/managed-settings/install.ts)

  writeFennecHooks(managedSettingsPath, hookCommandPath):
    1. Read existing file IF it exists (JSON.parse).
    2. If file doesn't exist OR is empty → start with { "hooks": {} }.
    3. For each of the 6 hooks (D-22), merge into hooks[<HookName>] array:
       { type: "command", command: hookCommandPath }
       — if an entry with the same command already exists, skip (idempotent install).
    4. Write back via fs.writeFileSync (atomic enough for managed-settings; concurrent writers not expected since this only runs at install time as root).
    5. chown root:wheel (uid 0, gid 0).
    6. chmod 644.
    Per Pitfall 2: NEVER use cp / overwrite. ALWAYS merge into the existing hooks object additively.

## Managed-settings uninstall (daemon/src/managed-settings/uninstall.ts)

  removeFennecHooks(managedSettingsPath, hookCommandPath):
    1. Read existing file. If not present → no-op return.
    2. For each hook name in hooks, FILTER out entries where command === hookCommandPath. Preserve all other entries (other tools' hooks, manually-added entries).
    3. If after filtering ALL hook arrays are empty AND no other top-level keys exist → fs.unlinkSync(path).
    4. Else write back the filtered JSON (preserve key order; 2-space indent per Pitfall 7).

## Path resolution (daemon/src/managed-settings/path.ts)

  resolveManagedSettingsPath(os: "darwin"|"linux"|"win32"): string
    darwin → /Library/Application Support/ClaudeCode/managed-settings.json
    linux → /etc/claude-code/managed-settings.json
    win32 → C:\\ProgramData\\ClaudeCode\\managed-settings.json (use process.env.ProgramData)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Claude Code → shim (process spawn) | Hook payload from Claude Code is parsed-only; shim never writes to filesystem |
| shim → daemon loopback | X-Fennec-Shim-Secret validates the caller; loopback is 127.0.0.1-only (never 0.0.0.0) |
| daemon loopback → adapter | In-process EventEmitter handoff; payload normaliser is the boundary that converts raw → CanonicalEvent shape |
| install script → managed-settings file (root) | Postinstall runs as root; file ACL pins to root:wheel mode 644 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing | Non-fennec process posts forged hook events to daemon loopback bridge | mitigate | Bridge requires X-Fennec-Shim-Secret header. Secret is read by daemon at boot from /etc/fennec/shim-secret (root-installed). Same-UID processes CAN read the secret (it's mode 0644), but that's accepted in Pattern 9's threat model — local loopback noise is the threat, not network attackers. |
| T-07-02 | Spoofing | Malicious binary planted at /usr/local/fennec/bin/fennec-hook to intercept hooks | mitigate | Postinstall script writes the shim binary with chown root:wheel mode 0755 (root-owned, user cannot replace without sudo). Plan 09 sets this. |
| T-07-03 | Denial of Service | Hook shim slow → Claude Code hot path delayed | mitigate | DAE-18: shim total budget 15ms. Go cold-start ~1ms. fail-open exits 0 within deadline. Unit test (`shim/main_test.go`) asserts the budget. Stress test: run 100 hook fires in sequence; total runtime <2s. |
| T-07-4 | Tampering | managed-settings.json overwritten on reinstall losing other tools' entries (Pitfall 2) | mitigate | install function additively merges. Test: pre-existing managed-settings.json with synapse-like entry → fennec install adds entries; synapse-like entry preserved. Uninstall removes ONLY fennec entries; synapse-like entry still present. |
| T-07-5 | Tampering | uninstall blasts away managed-settings.json content (Pitfall 7) | mitigate | Uninstall is surgical: filter fennec entries, preserve everything else. If file becomes empty (all empty arrays + no other keys), unlink it. Test asserts both: surgical removal AND empty-file deletion behavior. |
| T-07-6 | Tampering | User edits managed-settings.json to disable fennec hooks | mitigate | File is chmod 644 + chown root:wheel — user cannot edit without sudo. Per D-19. Verified by test that asserts file mode after install. |
| T-07-7 | Information Disclosure | Hook payload contains secrets, redactor in plan 06 didn't run yet | mitigate | Loopback bridge handoff → adapter.normalise → registry.emit IS the only path. registry.emit (plan 06) calls redactor.redact BEFORE queue.append. The shim itself never writes to disk. So secrets pass through (in memory) but never persist unredacted. Verified by canary smoke (plan 06) + integration test in this plan asserting a canary in a fake hook payload doesn't reach the JSONL queue. |
| T-07-SC | Tampering | New deps added by this plan: Go toolchain (system-installed, not npm) + no new npm deps | mitigate | The shim is a single-file Go program with stdlib-only imports (net/http, io, os, time, bytes). NO go modules used beyond stdlib. `shim/go.mod` declares zero external requires. |
