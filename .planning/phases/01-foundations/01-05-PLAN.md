## Routes (Hono)

POST /api/events/batch
  Auth: Authorization: Bearer <api_key>
  Body: EventBatchSchema (from @fennec/shared)
  Behavior:
    1. bearerAuth middleware resolves api_key → { org_id, api_key_id, daemon_machine_id } via sha256(token) lookup
    2. zValidator("json", EventBatchSchema) validates the batch
    3. For each event: parameterised UPSERT into ai_events with PK (idempotency_key, occurred_at), ON CONFLICT DO NOTHING
    4. Each row's org_id comes from auth context (NOT from request body)
    5. Returns { accepted: N }
    6. Hot path: NO correlation, NO model-fit, NO aggregation — these run as Phase 2 Queue consumers (Queue binding declared in wrangler.jsonc but no consumer in Phase 1)

POST /api/heartbeats
  Auth: Authorization: Bearer <api_key>
  Body: AdapterHeartbeatSchema (from @fennec/shared)
  Behavior: UPSERT into adapter_heartbeats with org_id + daemon_machine_id from auth context; idempotency_key UNIQUE prevents double-record

POST /api/daemons/enroll
  Auth: none (bootstrap endpoint)
  Body: EnrollRequestSchema (from @fennec/shared)
  Behavior:
    1. zValidator parses { install_secret, machine_id, hostname, os }
    2. Look up org by orgs.install_secret_hash = sha256(install_secret) AND orgs.install_secret_expires_at > NOW()
    3. If no org → 401 { error: "invalid_or_expired_install_secret" }
    4. UPSERT daemon_machines by (org_id, machine_id); returns the existing or new row
    5. If a non-revoked api_key already exists for this (org_id, daemon_machine_id) → return the existing api_key (idempotent re-enroll)
    6. Else generate a new bearer token (32-byte urandom, base64url), store sha256(token) in api_keys, return token
    7. Insert daemon_audit_events row with reason="enrollment_completed"
    8. Return EnrollResponseSchema { api_key, api_key_id, org_id, org_name, privacy_policy_url }

GET /api/auth/sso
  Auth: none
  Query: ?machine_id=<m>&redirect_uri=<loopback>&code_challenge=<S256>&state=<rand>&provider=<google|github|microsoft>
  Behavior: persist (state → { machine_id, redirect_uri, code_challenge, provider }) in a short-lived KV namespace (Workers KV binding) with 10-minute TTL, 302-redirect to OAuth provider authorize URL with state + the platform's actual client_id

POST /api/daemons/attach-callback
  Auth: none (PKCE verifier is the proof of session continuity)
  Body: AttachCallbackRequestSchema { code, state, code_verifier, machine_id }
  Behavior:
    1. zValidator parses input
    2. Look up state in KV → recover { machine_id, redirect_uri, code_challenge, provider }
    3. Verify base64url(sha256(code_verifier)) === code_challenge — if not, 400
    4. Exchange code + code_verifier for provider id_token via provider's /token endpoint
    5. Parse id_token to extract email
    6. UPSERT users by email; resolve user_id
    7. Insert org_members (org_id of the machine, user_id, role='member' default) ON CONFLICT DO NOTHING
    8. UPDATE daemon_machines SET attached_user_id = user_id, attached_at = NOW() WHERE id = (resolved by machine_id)
    9. Backfill: UPDATE ai_events SET user_id = <user_id>, user_id_unknown = NULL WHERE org_id = <org> AND hostname = <host> AND user_id IS NULL — one-shot per machine per D-15
    10. Insert daemon_audit_events row with reason="attach_completed"
    11. Return AttachCallbackResponseSchema { user_id, email, org_id }

POST /api/daemons/uninstall
  Auth: Authorization: Bearer <api_key>
  Body: UninstallAuditEventSchema (from @fennec/shared)
  Behavior:
    1. bearerAuth resolves api_key → { org_id, daemon_machine_id }
    2. zValidator parses input
    3. Insert daemon_audit_events with reason (from input; must match user_initiated|mdm_revoke|admin_initiated)
    4. UPDATE api_keys SET revoked_at = NOW() WHERE id = api_key_id
    5. Returns { audit_id }

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| daemon → backend | TLS terminates at Cloudflare; Bearer token is the only credential; `org_id` is derived server-side from `api_keys.token_hash` lookup, NEVER read from request body |
| browser (OAuth provider) → backend `/attach-callback` | PKCE verifier proves session continuity; `state` parameter prevents CSRF; code exchange happens server-side with provider |
| Worker → Postgres via Hyperdrive | TLS-encrypted; `service_role` JWT bypasses RLS (Phase 1); RLS becomes load-bearing in Phase 3 |
| `install_secret` over the wire | TLS protects in transit; sha256-hashed server-side at lookup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | Forged Bearer token | mitigate | `resolveApiKey` looks up `sha256(token)` in `api_keys.token_hash WHERE revoked_at IS NULL`; mismatch → 401. Token entropy is 32 bytes urandom → 2^256 keyspace. |
| T-05-02 | Tampering | Client supplies `org_id` in event payload to leak across tenants | mitigate | `org_id` is NEVER read from request body. Backend stamps `org_id` from auth context (`c.get("org_id")`) before INSERT. Per Pattern 11 in 01-RESEARCH.md. Integration test asserts even a hostile `events[i].org_id` field in the payload is ignored. |
| T-05-03 | Tampering | SQL injection via `install_secret` lookup or `idempotency_key` upsert | mitigate | All queries use `pg` parameterised statements (`$1`, `$2`, ...). NO string concatenation of values into SQL. Verified via grep in integration tests. |
| T-05-04 | Information Disclosure | Bearer token leaks via daemon log → backend log | mitigate | Backend logs the api_key_id (UUID) NOT the raw token. Unit test asserts no log call references `c.req.header("authorization")`. |
| T-05-05 | Repudiation | Daemon enrolls, sends events, then claims it didn't | mitigate | `daemon_audit_events` records enrollment + attach + uninstall with reason + actor + timestamp; trail is permanent (no DELETE in Phase 1). |
| T-05-06 | Tampering | Replay of `install_secret` against multiple machines to enroll a malicious daemon | mitigate | Each enrollment binds (org_id, machine_id) → daemon_machines (UNIQUE constraint). Idempotent re-enrollment returns the SAME api_key for the same machine_id; a malicious actor would need both a valid install_secret AND a NEW machine_id (controllable but enrolled machine count is auditable per Phase 4 dashboard). Install_secret rotation policy = "support-only" per Open Question 2; expires_at default 90 days (set in plan 04 schema). |
| T-05-07 | Spoofing | OAuth state parameter CSRF | mitigate | `state` is stored in KV at /api/auth/sso, looked up at /attach-callback. Mismatched or missing state → 400. KV TTL = 10 min. |
| T-05-08 | Tampering | OAuth `code` intercepted by malicious local process | mitigate | PKCE verifier mandatory per RFC 8252. Even if code is intercepted, the malicious process lacks the verifier (kept in daemon memory only). Loopback redirect URI per Pattern 10. |
| T-05-09 | Elevation of Privilege | Backend uses service_role; if Worker is compromised, all data is exposed | accept | Phase 1 limitation — service_role bypasses RLS. Mitigation: Worker is minimum-privileged at the network edge; secrets via Wrangler secrets (encrypted at rest); Phase 3 introduces user-scoped tokens + RLS becomes load-bearing. |
| T-05-10 | Information Disclosure | Cross-tenant data leak via missing middleware check | mitigate | `org_id` derived from auth context on every route. Phase 3 tenant-isolation drill (per 01-RESEARCH.md §Anti-Patterns) tests this; Phase 1 unit test on `events-batch.test.ts` asserts request-body `org_id` is ignored. |
| T-05-SC | Tampering | npm installs of new backend deps (hono, @hono/zod-validator, @supabase/supabase-js, pg) | mitigate | All packages already verified at the Plan 01 Task 1 slopcheck human checkpoint; this plan uses the EXACT same versions listed in 01-RESEARCH.md §Package Legitimacy Audit. If a new dep is needed, halt and add to audit + re-checkpoint. |
