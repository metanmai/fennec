<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dev machine → Supabase project (db push) | SUPABASE_ACCESS_TOKEN authenticates the dev to the project; tokens are env-vars only |
| dev machine → Cloudflare Workers (wrangler deploy) | CLOUDFLARE_API_TOKEN authenticates the dev to the account |
| installed daemon → backend Worker | Bearer token (api_key) per machine |
| signed .pkg install → macOS Gatekeeper | Apple notarisation ticket validated before postinstall executes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | supabase db push applies a tampered migration | mitigate | Migrations are in version control; SHA-256 of migration directory recorded in 01-SMOKE-LOG.md before push; supabase CLI runs only the files in supabase/migrations/. |
| T-10-02 | Information Disclosure | SUPABASE_ACCESS_TOKEN or CLOUDFLARE_API_TOKEN leaks via shell history | mitigate | Tokens set via env var in a single shell session; documented in deploy script + 01-SMOKE-LOG.md as "set via env, not committed". `.gitignore` excludes any `.env*` files (added in plan 01). |
| T-10-03 | Information Disclosure | Smoke test inserts known test plaintexts into a production Supabase project | mitigate | Test plaintexts (FENNEC_TEST_INSTALL_SECRET_PHASE1_... and fennec_phase1_smoke_TESTKEY_...) are seeded by migration 7 which is tagged DEV-ONLY. The Supabase project used here is the Phase 1 development project — never a customer-facing production. Phase 6 deployment docs explicitly tell operators to skip migration 7. |
| T-10-04 | Denial of Service | Failed `supabase db push` leaves schema in a partial state | mitigate | supabase CLI uses transactional migrations by default; failure → rollback. If a migration is non-rollbackable (rare; we don't use any), document recovery in `01-SMOKE-LOG.md` and run `supabase db reset` for the dev project. |
| T-10-05 | Tampering | Smoke test uses a different .pkg than the signed one | mitigate | scripts/db-push.sh logs the SHA-256 of installer/build/fennec.pkg before any test; spctl --assess re-runs at install time and must report `source=Notarized Developer ID`. |

</threat_model>
