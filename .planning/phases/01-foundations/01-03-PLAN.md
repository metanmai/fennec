<objective>
Procure the external code-signing assets that gate Phase 1's distribution (DAE-08 macOS notarisation, DAE-09 Windows EV cert clock-start) and verify both signing pipelines work on test artefacts. Per D-05, Phase 1 CANNOT ship without an Apple-notarised .pkg, and the Windows EV cert reputation-warm-up clock starts at first signature — so this plan runs in parallel with schema work to unblock plan 09.

Purpose: Code-signing certs are external dependencies with multi-day lead times (Apple Developer Program enrollment up to 24 hours; Windows EV cert 2-7 days per A4). The clock cannot be started later. The certificate-procurement playbook also belongs to fennec because the procurement decision (vendor, cost, validity period) must be tracked alongside the code.

This plan does NOT build any product feature. It buys credentials, generates Apple's API key, configures keychain credentials, signs a throwaway test artefact, and records every relevant ID + thumbprint + timestamp for plan 09 to consume.

Output: Apple Developer Program enrollment active; Developer ID Installer cert in keychain; App Store Connect API key in keychain via `xcrun notarytool store-credentials "fennec-notary"`; Windows EV cert + HSM/token in hand; test artefact signed with both; status tracked in `.planning/phases/01-foundations/01-CERT-STATUS.md`.
</objective>

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer machine → Apple Developer portal | Developer ID cert private key created in keychain; never leaves the signing machine |
| developer machine → Apple notary service | Notarytool API key authenticates the dev machine to Apple's notary; tied to the developer account |
| HSM/token → developer machine | Windows EV cert private key is hardware-bound (FIPS 140-2 Level 2+ requirement for EV); cannot be exported |
| developer machine → certificate vendor | ID-verification documents (passport, business docs) submitted; vendor MUST be reputable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Developer ID private key extracted from keychain | mitigate | Apple keychain marks Developer ID private keys non-exportable by default; user MUST NOT enable export. macOS FileVault adds at-rest disk encryption. |
| T-03-02 | Tampering | Windows EV cert private key extracted from HSM | mitigate | EV cert HSMs are FIPS 140-2 Level 2 minimum (vendor requirement); private key is unextractable. Token PIN never written to disk; reset on suspicion. |
| T-03-03 | Information Disclosure | App Store Connect API key (`.p8` file) committed to git | mitigate | `.p8` file stored OUTSIDE the repo (in `~/.config/fennec-keys/` or similar); `xcrun notarytool store-credentials` imports it into keychain, then the `.p8` can be moved to offline storage. `.gitignore` already excludes any `*.p8` files. |
| T-03-04 | Spoofing | Vendor mis-identifies the developer and issues an EV cert to the wrong party | mitigate | Buy from a CA/B Forum member (DigiCert, Sectigo, Certera all qualify per 01-RESEARCH.md); ID-verification process is multi-step. |
| T-03-05 | Tampering | Counterfeit signing token or fake "EV cert" purchased from non-CA vendor | mitigate | Only buy from CA/B Forum members listed in `installer/windows/CERT-PROCUREMENT.md`. Never buy "EV certs" from marketplaces or unknown brokers. |
| T-03-SC | Tampering | No npm/pip/cargo installs in this plan | n/a | This plan does not install packages — supply chain considerations only apply to the test artefact, which is a trivial Go program or pre-existing binary |

</threat_model>
