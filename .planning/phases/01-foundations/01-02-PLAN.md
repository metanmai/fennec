<objective>
Freeze the canonical event schema and authentication request shapes in `@fennec/shared` so daemon (Node 22) and backend (Cloudflare Workers) consume an identical, Zod-validated, type-safe wire format. This package is runtime-neutral (Zod only, no `node:*` imports) so Workers can import it without polyfills.

Purpose: Wire-format drift between daemon and backend is the most expensive bug class in synapse's history (per 01-RESEARCH.md). A single Zod source of truth prevents that. Phase 1's six requirements derived from this package — CAP-10 (canonical schema), CAP-13 (idempotency key), ANL-06 (separate cache token fields) — all consume the artifacts here. Plans 05 and 06 (backend + daemon) cannot start until this contract is locked.

Output: `@fennec/shared` exports `CanonicalEventSchema`, `EventBatchSchema`, `ClaudeCodePromptPayloadSchema`, `AdapterHeartbeatSchema`, `deriveIdempotencyKey()`, `EnrollRequestSchema`, `EnrollResponseSchema`, `AttachCallbackRequestSchema`, `UninstallAuditEventSchema`. Vitest unit tests prove (1) malformed payloads are rejected with clear errors, (2) idempotency keys are stable across retries, (3) cache token fields stay separated.
</objective>

From 01-RESEARCH.md §Canonical Event Schema (Pattern-1 source-of-truth):

EventKind discriminated union:
  - prompt_submitted
  - tool_call
  - session_start
  - session_end
  - pre_compact
  - subagent_stop
  - model_response

CanonicalEvent top-level fields (universal across adapters):
  - idempotency_key: string (>=1 chars, sha256-hex of 32 chars in practice)
  - tool: enum("claude-code","codex","gemini","cursor","copilot","chatgpt-web","claude-ai-web","git")
  - adapter_version: string
  - occurred_at: ISO datetime string
  - cwd: optional string
  - git_remote: optional string
  - git_branch: optional string
  - hostname: string
  - os: enum("darwin","linux","win32")
  - kind: EventKind
  - payload: Record<string, unknown> (adapter-specific; validated per kind in adapter)
  - schema_version: literal 1
  - redaction_applied_at: ISO datetime string
  - redaction_version_hash: string

ClaudeCodePromptPayload fields:
  - prompt_text: string (POST-redaction)
  - session_id: string
  - cwd: optional string
  - hook_event: string ("UserPromptSubmit","PostToolUse","SessionStart","SessionEnd","PreCompact","SubagentStop")
  - usage: optional { input_tokens?, output_tokens?, cache_creation_input_tokens?, cache_read_input_tokens? } — all four non-negative ints, SEPARATE

EventBatch:
  - events: array of CanonicalEvent, min 1, max 500

AdapterHeartbeat:
  - idempotency_key: string (hostname|adapter|interval_start)
  - hostname: string
  - adapter: string
  - adapter_version: string
  - schema_hash: string (per-adapter source-tool data shape fingerprint, planner choice = field-name set hash per Open Question 3)
  - events_parsed: non-negative int
  - parse_errors: non-negative int
  - daemon_unreachable_count: non-negative int, default 0
  - interval_start: ISO datetime
  - interval_end: ISO datetime
  - schema_version: literal 1

EnrollRequest:
  - install_secret: string min 32 chars
  - machine_id: string min 8 chars
  - hostname: string
  - os: enum("darwin","linux","win32")

EnrollResponse:
  - api_key: string
  - api_key_id: string (uuid)
  - org_id: string (uuid)
  - org_name: string
  - privacy_policy_url: string (URL)

AttachCallbackRequest (OAuth PKCE per RFC 8252 §7.3 + Pattern 10):
  - code: string (OAuth authorization code)
  - state: string
  - code_verifier: string (PKCE verifier, base64url, 43-128 chars)
  - machine_id: string

UninstallAuditEvent (D-18, D-19):
  - idempotency_key: string
  - machine_id: string
  - hostname: string
  - reason: enum("user_initiated","mdm_revoke","admin_initiated")
  - actor: optional string (org admin email or local user)
  - occurred_at: ISO datetime
  - schema_version: literal 1

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| daemon → backend | CanonicalEvent + EventBatch crosses HTTPS; both sides re-validate via the same Zod schema |
| shim → daemon | Hook payload (Anthropic-emitted JSON) enters as untrusted; daemon parses with Zod before adding to queue |
| installer → daemon | EnrollRequest contains install_secret (must validate length >=32 to enforce entropy) |
| OAuth provider → daemon | AttachCallbackRequest has code+state+verifier; backend re-validates with Zod before exchanging |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | CanonicalEvent payload manipulated in transit | mitigate | TLS at the transport layer (plan 05 backend) + Zod schema rejection at backend ingress + `org_id` server-side stamped (not client-supplied; per Pattern 11 in 01-RESEARCH.md, `org_id` is set from api_key lookup, never read from request body) |
| T-02-02 | Spoofing | Malicious client supplies forged `idempotency_key` to overwrite legitimate event | mitigate | Backend uses `ON CONFLICT (idempotency_key) DO NOTHING` (plan 04 schema) — duplicate keys are no-ops; original event preserved. Schema enforces minimum entropy of the key (32-hex). |
| T-02-03 | Information Disclosure | Cache token fields aggregated at capture, producing 70% miscount (LiteLLM bug pattern per Pitfall 6) | mitigate | All 4 token fields (input/output/cache_creation/cache_read) are SEPARATE optional fields on `usage`. NEVER add them at capture. Unit test asserts the four-field structure cannot be collapsed. |
| T-02-04 | Tampering | `install_secret` brute-forced via short keyspace | mitigate | Zod enforces min 32 chars on `EnrollRequestSchema.install_secret`. Backend (plan 05) hashes the secret server-side; cannot be brute-forced via the API alone (rate limiting added in Phase 3). |
| T-02-05 | Information Disclosure | Cache token semantics misinterpreted (Assumption A2 — Anthropic vs OTel disagreement) | mitigate | Capture all 4 fields RAW + VERBATIM. Phase 1 daemon NEVER computes totals. Cost computation deferred to Phase 2 worker where it can be fixed without redeploying daemons. Cross-referenced as `checkpoint:human-verify` in plan 06 (daemon redaction wires). |

</threat_model>
