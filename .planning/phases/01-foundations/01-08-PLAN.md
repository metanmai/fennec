## Enrollment client (daemon/src/enroll/enroll.ts)

  enrollDaemon({ installSecret, machineId, hostname, os, apiBaseUrl, fetchOpts? }): Promise<EnrollResponse>
    - Build body = { install_secret, machine_id, hostname, os }
    - Validate locally via EnrollRequestSchema (from @fennec/shared)
    - POST ${apiBaseUrl}/api/daemons/enroll (no auth header)
    - Parse response via EnrollResponseSchema
    - Return { api_key, api_key_id, org_id, org_name, privacy_policy_url }
    - On 401: throw error "invalid_or_expired_install_secret"
    - On 4xx/5xx: throw with reason

## API key store (daemon/src/enroll/api-key-store.ts) — AUTH-15

  KEY_PATH on darwin: /var/db/fennec/key (mode 0400, owner root)
  KEY_PATH on linux: /var/lib/fennec/key (Phase 5; daemon code supports it for future-portability)
  KEY_PATH on win32: %ProgramData%\\fennec\\key (Phase 5)

  persistApiKey(apiKey: string, os: "darwin"|"linux"|"win32"): void
    1. Resolve KEY_PATH per OS
    2. mkdirSync(parent, { recursive: true, mode: 0o700 })
    3. writeFileSync(KEY_PATH, apiKey, { encoding: "utf-8", mode: 0o400 })
    4. chmodSync(KEY_PATH, 0o400) — belt-and-suspenders
    5. if process.getuid?.() === 0 → chownSync(KEY_PATH, 0, 0) (root:wheel on macOS, root:root on Linux)
    6. assertSafeKeyPath(KEY_PATH) — post-write verification

  readApiKey(os): string
    1. statSync(KEY_PATH)
    2. if (mode & 0o777) !== 0o400 → throw "api-key-file-permissions-drifted" — daemon refuses to use the key
    3. if uid !== 0 → throw "api-key-file-not-root-owned" — daemon refuses to use the key
    4. return readFileSync(KEY_PATH, "utf-8").trim()

  assertSafeKeyPath(path): void — internal helper, runs at install + every read

  Per Pitfall 10: NEVER cache the result. Every read re-checks permissions.

## Machine ID (daemon/src/enroll/machine-id.ts) — D-13

  getMachineId(os): string
    darwin: exec `ioreg -rd1 -c IOPlatformExpertDevice | grep IOPlatformUUID | awk -F'"' '{print $4}'` (or use IOKit FFI; for Phase 1 the shell-out is fine)
    linux: read /etc/machine-id (systemd) or /var/lib/dbus/machine-id (fallback) — Phase 5 concern, but skeleton exists today
    win32: read HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Cryptography\MachineGuid — Phase 5 concern

  Phase 1 only needs darwin; linux + win32 return placeholder values that throw "machine_id_unsupported_on_phase1_os" if invoked on those OS — explicit failure rather than silent wrong-value.

## PKCE (daemon/src/attach/pkce.ts) — Pattern 10

  generatePkcePair(): Promise<{ code_verifier: string, code_challenge: string }>
    - code_verifier: 32 bytes crypto.getRandomValues → base64url-encode (43 chars per RFC 7636 §4.1)
    - code_challenge: base64url(sha256(code_verifier)) using Web Crypto API

## One-shot OAuth callback server (daemon/src/attach/oauth-server.ts)

  class OneShotOAuthServer
    Constructor: {} (no args)
    start(): Promise<{ port: number, callbackUrl: string, awaitCode: Promise<{ code: string, state: string }> }>
      - http.createServer(...).listen(0, "127.0.0.1") — random free port
      - Handles GET /callback?code=...&state=... → resolves the awaitCode promise + responds 200 "Sign-in complete. You can close this tab."
      - Handles any other path → 404
      - Auto-stops after the promise resolves OR after 5-minute timeout (which rejects awaitCode)
    stop(): void

## Notifier bridge (daemon/src/attach/notifier-bridge.ts)

  class NotifierBridge
    Constructor: { notifierPort: number = 7822 } — different from daemon's main loopback bridge port (7821)
    notify({ title, message, openUrl?: string }): Promise<{ delivered: boolean }>
      - POST http://127.0.0.1:${notifierPort}/v1/notify with body
      - If POST fails (notifier not running): { delivered: false } — does NOT throw (fail-open per Pattern 6)
      - If 2xx: { delivered: true }
    Per A5 mitigation: the notifier IS the Helper LaunchAgent that runs in user-session; we cannot call osascript directly from the LaunchDaemon. If the user has logged out, the notifier process is gone and POST fails — that's acceptable, daemon continues operating.

## Notifier binary (notifier/main.go)

  Tiny Go program (~80 LOC). Builds a HTTP server bound to 127.0.0.1:${FENNEC_NOTIFIER_PORT:-7822}:
  Routes:
    POST /v1/notify
      Body: { title: string, message: string, openUrl?: string }
      Behavior:
        - exec osascript -e `display notification "<message>" with title "<title>"` (macOS)
        - if openUrl present: exec `open "${openUrl}"` (macOS) — opens default browser
        - return 200 { delivered: true }
      For linux/win32: stub — return 200 + log "platform_not_supported_phase1"
    GET /v1/health
      Return 200 { status: "ok" }
  Cross-compile via notifier/Makefile (mirror shim/Makefile).

## Attach flow (daemon/src/attach/attach.ts)

  runAttachFlow({ apiBaseUrl, machineId, provider, notifier }): Promise<{ user_id, email, org_id }>
    1. Generate PKCE pair
    2. Start OneShotOAuthServer; get port + callbackUrl
    3. Compute state = crypto.randomUUID()
    4. Build sso_url = `${apiBaseUrl}/api/auth/sso?machine_id=${machineId}&redirect_uri=${callbackUrl}&code_challenge=${code_challenge}&state=${state}&provider=${provider}`
    5. notifier.notify({ title: "Sign in to fennec", message: "Click to attribute your AI usage to your developer identity", openUrl: sso_url })
       If notifier returns delivered=false: log warning, still wait on awaitCode (user might paste URL manually if they had a previous browser open)
    6. await server.awaitCode (with 5-minute timeout)
    7. POST ${apiBaseUrl}/api/daemons/attach-callback with { code, state, code_verifier, machine_id }
    8. Parse AttachCallbackResponseSchema → return { user_id, email, org_id }

  On any throw: daemon continues operating with events tagged unknown@${hostname}; D-15 says backfill happens on first successful attach.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MDM payload / wizard input → daemon enrollment | install_secret is plaintext in transit (TLS); validated server-side via sha256 lookup |
| daemon → /var/db/fennec/key (root-only) | Filesystem ACL is the only barrier between daemon's api_key and rogue local processes |
| daemon → notifier (loopback) | Notifier runs in user-session (LaunchAgent); communication via 127.0.0.1; no secret on this channel since the notifier's job is purely UX |
| OAuth provider → loopback redirect URI | PKCE verifier prevents code interception per RFC 8252 |
